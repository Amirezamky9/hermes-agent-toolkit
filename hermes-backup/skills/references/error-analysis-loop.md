# Error Analysis Loop — n8n Builder Quality Improvement

## The Pattern: Judge + Worker + Analysis Agent

When LLMs generate n8n workflow code, errors happen. This reference captures a **closed-loop system** for catching, logging, analyzing, and preventing repeated errors.

The system has **three roles** and **two data flows** (per-session and cumulative):

### Three roles

| Role | Who | When | What |
|------|-----|------|------|
| 👨‍⚖️ **Judge** | Hermes (primary agent) | **During session** — log observations, model mistakes, oversights. **Morning** — read nightly report, patch skills/trap-ref, report to user | |
| 🌙 **Worker** | Cron job (2 AM) + nightly script | Nightly | Scans session_search, reads session-logs, builds nightly report |
| 🤖 **Analysis Agent** | Sub-agent (weekly/monthly) | Weekly/Monthly | Reads accumulated session-logs + reports, finds patterns, suggests structural fixes |

### Two data flows

```
PER-SESSION FLOW (every session, both roles write):
  👤 User (end) + 👨‍⚖️ Judge (during)  →  session-logs/YYYY-MM-DD_session-log.yaml
  👤 User (ad-hoc)                      →  bugs/manual-fixes.yaml (for bugs user fixed manually)
  👤 User (ad-hoc)                      →  feedback/YYYY-MM-DD_feedback.yaml (corrections mid-conversation)

NIGHTLY FLOW (2 AM cron):
  🌙 Worker script    → runs nightly_worker.py → reads today's files
  🌙 Worker agent     → session_search + cross-reference trap-ref → reports/YYYY-MM-DD_report.yaml

MORNING FLOW:
  👨‍⚖️ Judge reads report → patches trap-ref → patches SKILL → reports to user
```

## 📋 Session Log Schema (per-session, both write)

### What gets written

| Who writes | Section | Content |
|-----------|---------|---------|
| 👤 **User** (at end of session) | `user_manual_fixes` | Bugs they fixed manually that the model couldn't |
| | `user_feedback` | Corrections / preferences / style notes |
| | `user_suggestions` | Ideas, improvement proposals |
| | `user_missed` | Things the user felt the agent misunderstood or overlooked |
| 🤖 **Agent (Judge, during session)** | `judge_model_mistakes` | Model errors caught mid‑work |
| | `judge_observations` | Oversights, missed requirements, edge cases |
| | `judge_stats` | Token usage, success/fail counts |

### YAML schema

```yaml
session:
  id: "dc4a47b71add"               # from /status or session_search
  date: "2026-07-09"
  phase: "builder"                  # intake | planner | reviewer | builder
  model_used: "opencode200k"
  topic: "WF01 Cart Module"

user_manual_fixes:                  # 🛠️ Bugs user fixed manually
  - symptom: "DataTable columns.value خالی میاد"
    manual_fix: "رفتم توی UI n8n و دستی ست کردم"
    root_cause: "addNode پارامتر رو میندازه"
    n8n_ui_fix: true

user_feedback:                      # 💡 Corrections / preferences
  - topic: "Switch routing"
    detail: "گفتم از Execute Sub-workflow استفاده کن نه HTTP Request"
    type: "correction | improvement | best_practice"

user_suggestions:                   # 💭 Ideas
  - "بیا یه ماژول خطایابی جداگانه بسازیم"

user_missed:                        # ⚠️ Things felt missed
  - "حس کردم متوجه نشدی که session state باید normalize بشه"

judge_model_mistakes:               # 🎯 Model errors caught
  - detail: "مدل دوباره update زد بجای upsert"
    model: "opencode200k"
    in_trap_ref: false

judge_observations:                 # 🔍 Things agent noticed
  - detail: "کاربر ۱۰ بار /help زد — هندل نشد"
    priority: medium

judge_stats:
  successful_nodes: 0
  failed_attempts: 0
  tokens_used: 0
```

## 🧠 Error Categorization Schema

Every entry in a nightly report gets categorized:

```
ERROR TYPE         │  WHAT IT LOOKS LIKE           │  TRAP-REF SECTION
───────────────────┼───────────────────────────────┼────────────────────
Schema/Syntax      │  invalid param name, missing  │  Node Parameters
Connection         │  wires go to wrong output     │  Wiring
Type Mismatch      │  number vs string on Switch   │  Type Validation
Credential         │  wrong token, missing ID      │  Credentials
Silent No-Op       │  upsert returns 0 rows        │  DataTable / Traps
Truncation         │  >20 nodes, SDK truncates     │  Node Count Limits
Duplicate Execution   │  node runs 2+ times with same input  │  Converging Branches
Credential Namespace  │  v2 vs v2.6 Postgres, same name     │  Postgres Version
```

## 🌙 Nightly Worker — Implemented

The nightly pipeline is **already deployed** on this instance:

```yaml
Cron Job:    n8n-nightly-worker (job_id: 78742e51edf2)
Schedule:    0 2 * * * (every night at 2:00 AM UTC)
Script:      ~/.hermes/scripts/nightly_worker.py
Deliver:     origin (this Telegram DM topic)
Repeat:      forever
```

### Script (`nightly_worker.py`)

The script (`~/.hermes/scripts/nightly_worker.py`, symlinked from `/workspace/n8n-wfb/scripts/nightly_worker.py`):

1. Reads today's files from `session-logs/`, `feedback/`, `bugs/manual-fixes.yaml`
2. Merges into a YAML report
3. Saves to `reports/YYYY-MM-DD_report.yaml`
4. Prints summary for cron delivery

Then the **Worker agent** (the LLM running the cron prompt) takes over:

1. Reads the generated report
2. Runs `session_search` for "error|bug|fix|mistake|trap" in the last 24h
3. Cross-references against `trap-reference.md` and `bugs/platform-bugs.md`
4. Outputs a PRSian summary: what was found, what's new, what's repeated
5. Flags **"از چشم ما افتاد"** items — things BOTH user and judge missed, caught by the worker's broader search

### Cron creation reference

```bash
hermes cron create                        \
  --name "n8n-nightly-worker"             \
  --schedule "0 2 * * *"                  \
  --script "nightly_worker.py"            \
  --prompt "NIGHTLY REPORT: run script, session_search, cross-ref trap-ref, output Persian summary"
```

## ☀️ Morning Report (Judge)

After the worker runs, the judge reads the log and:

1. **Acknowledge** — "📊 دیشب N تا خطا، M تا جدید، K تا دستی رفع شد"
2. **Patch trap-ref** — add new errors to `trap-reference.md`
3. **Patch skill** — if a pattern warrants it, update `n8n-builder/SKILL.md`
4. **Flag "از چشم ما افتاد" items** — things both missed that the worker caught by cross-session analysis
5. **Report** — tell the user what was learned + what was fixed

## 📊 Weekly Analysis Agent

Every 7-14 days, a sub-agent reads accumulated reports:

```
Input: /workspace/n8n-wfb/reports/ for the last 30 days
Task: Categorize, find patterns, rank by frequency, propose structural fixes
Output: Improvement recommendations
  e.g. "50% of errors are DataTable param drops → need a preprocessing wrapper"
  e.g. "30% are Switch type mismatches → add a type-safe normalization step"
```

## 🗺️ Product Roadmap (3 Phases)

This system evolves in three phases:

| Phase | What | Status |
|-------|------|--------|
| **1. Bootstrapping** | File structure, templates, nightly script, cron job, manual logging | ✅ **DONE** |
| **2. Automation** | Auto-analyze with sub-agent, auto-patch trap-ref, trend dashboard | 🔜 Next |
| **3. Product Maturity** | Fixer agent, auto-rollback, SLA monitor, error budget | 🔮 Future |

---

## 🚫 When to Skip (do NOT log these)

- One-time environment errors (disk full, network blip, server restart)
- Known transient n8n platform bugs (already in trap-ref)
- Errors from intentional user experimentation
